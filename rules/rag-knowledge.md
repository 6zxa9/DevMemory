# RAG Knowledge Base — автоматический обмен знаниями

## Обязательные правила (работают в КАЖДОМ проекте)

### Перед реализацией любой нетривиальной задачи:

1. Запросить Pinecone (ключевые слова на русском И английском):
   ```bash
   python ~/.claude/tools/pinecone-store.py query "<описание задачи EN + RU>" --ns all --top-k 10 -v
   ```
2. Если найдено (score > 0.6):
   - Показать пользователю найденное решение с кодом
   - Проверить поле `verified_libs` — если версия библиотеки устарела, предупредить
   - Проверить поле `ttl_check_after` — если дата прошла, пометить как "требует ревалидации"
   - Проверить актуальность через Context7 (resolve-library-id → query-docs)
   - Сравнить: решение из базы vs актуальная документация
   - Предложить выбор: база / документация / гибрид
3. Если не найдено — кратко сообщить и продолжить

### После успешного решения:

1. Если пользователь говорит **"отлично"** / **"запиши"** / **"сохрани"** → ОБЯЗАТЕЛЬНО предложить сохранить
2. Если завершена нетривиальная фича (3+ файлов) → предложить сохранить
3. Формат записи: описание (EN + RU) + реальный код (20-80 строк) + WHY + REJECTED
4. Обязательные metadata поля:
   - `description_ru` — русское описание (дублирует description)
   - `verified_libs` — библиотеки с версиями (например "drizzle-orm@0.30, react@19")
   - `verified_date` — дата проверки (YYYY-MM-DD)
   - `ttl_check_after` — дата ревалидации (+6 месяцев от verified_date)
5. Загрузить:
   ```bash
   python ~/.claude/tools/pinecone-store.py store-patterns /tmp/rag_entry.json
   # или для багов:
   python ~/.claude/tools/pinecone-store.py store-lessons /tmp/rag_lesson.json
   ```

### Context7 сравнение (при каждом READ):

1. Определить библиотеки в найденном решении (из поля `verified_libs`)
2. resolve-library-id → query-docs для проверки актуальности
3. Если API изменился или есть лучший подход → предупредить
4. Если `ttl_check_after` прошла → обязательно проверить через Context7
5. Не копировать слепо — решение из базы = reference, не template

### Что сохранять:

- Решения с конкретным кодом, проверенные в production
- Неочевидные workaround'ы платформ
- Формулы и числа, найденные через итерации
- Архитектурные решения с обоснованием и отвергнутыми альтернативами
- Production баги с root cause и prevention rule

### Что НЕ сохранять:

- Очевидные вещи из документации
- Код без контекста "почему"
- Промежуточные/непроверенные решения
